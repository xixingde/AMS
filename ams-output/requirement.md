# 系统需求规格说明书

---

**需求质量评估**：
- 最终得分：88
- 理由：功能清单完整，数据结构与字段规则详尽，权限与行为边界经澄清后已明确，足以支撑架构设计与开发拆解。

## 一、概述

### 1.1 项目背景

为规范企业 IT 系统的架构资产管理，实现系统、子系统、部署子系统、发布单元及技术栈的统一维护与可视化展示，建设架构管理系统（AMS）。系统支持通过 Web 管理后台进行架构元数据的增删改查、Excel 批量导入导出，以及基于 Mermaid 格式的逻辑部署架构图展示与发布单元调用关系管理。

### 1.2 参考文档

- 《架构管理系统需求_1.md》（用户原始需求文档）

### 1.3 用户角色

| 角色 | 说明 |
| :--- | :--- |
| **超级管理员** | 可设置管理员，并拥有系统所有权限 |
| **管理员** | 可对系统清单、技术栈、逻辑部署架构图进行增删改查 |
| **普通用户** | 可查询系统清单、技术栈信息、子系统发布单元、子系统逻辑部署架构图 |

## 二、用户故事

### US-001 - 系统清单管理

作为管理员，我希望对系统、子系统、部署子系统、发布单元进行增删改查和 Excel 导入导出，以便统一维护企业 IT 架构资产。

**验收场景**：
1. **Given** 管理员已登录系统，**When** 进入系统清单管理页面并填写系统信息，**Then** 系统保存该系统记录并支持按编号、名称、类型、属主、级别等字段查询。
2. **Given** 管理员在系统列表页面，**When** 选择一条系统记录并点击编辑，**Then** 系统展示编辑表单，保存后更新记录，并维持子系统、部署子系统、发布单元的层级关系。
3. **Given** 管理员在系统列表页面，**When** 选择一条存在下级子系统的系统记录并点击删除，**Then** 系统禁止删除并提示“请先删除或迁移下级子系统后再执行删除操作”。
4. **Given** 管理员在系统清单页面，**When** 上传符合模板格式的 Excel 文件，**Then** 系统解析并导入数据；若数据已存在（按唯一编号判断），则覆盖更新，并在导入完成后返回导入结果摘要。
5. **Given** 管理员在系统清单页面已设置查询条件，**When** 点击“导出 Excel”，**Then** 系统按当前查询条件过滤结果并导出为 Excel 文件。

### US-002 - 技术栈管理

作为管理员，我希望对技术栈进行增删改查和 Excel 导入导出，以便统一维护企业技术选型规范。

**验收场景**：
1. **Given** 管理员已登录系统，**When** 进入技术栈管理页面并填写技术栈分类、名称、选型建议、负责条线等信息，**Then** 系统保存该技术栈记录。
2. **Given** 管理员在技术栈列表页面，**When** 通过技术栈编号、分类、名称、选型建议、负责条线等字段进行查询，**Then** 系统返回匹配的技术栈列表。
3. **Given** 管理员在技术栈管理页面，**When** 上传符合模板格式的 Excel 文件，**Then** 系统导入技术栈数据，已存在数据按唯一编号覆盖更新。
4. **Given** 管理员已设置查询条件，**When** 点击“导出 Excel”，**Then** 系统按当前查询条件导出技术栈数据。

### US-003 - 逻辑部署架构图管理

作为普通用户或管理员，我希望查看部署子系统的逻辑部署架构图，以便直观了解系统在网络区域中的部署结构和调用关系。

**验收场景**：
1. **Given** 用户进入某个部署子系统的详情页面，**When** 查看逻辑部署架构图，**Then** 系统基于该部署子系统下的发布单元、网络区域及调用关系自动生成 Mermaid 格式的架构图，图中展示网络区域、发布单元名称及调用关系。
2. **Given** 管理员在架构图编辑模式，**When** 手动调整节点位置或增加备注说明，**Then** 系统保存手动调整后的布局与备注，后续展示时优先使用已保存的手动调整结果。
3. **Given** 架构图中存在当前系统发布单元和其它系统发布单元，**When** 渲染架构图时，**Then** 当前系统发布单元以绿色（#90EE90）标识，其它系统发布单元以浅绿色（#E6F7E6）标识。

### US-004 - 发布单元调用关系管理

作为管理员，我希望对发布单元之间的调用关系进行增删改查，以便维护系统间的依赖与通信拓扑。

**验收场景**：
1. **Given** 管理员已登录系统，**When** 进入调用关系管理页面并选择部署子系统、调用发布单元、被调用发布单元及调用协议（如 HTTP、TCP 等），**Then** 系统保存该调用关系记录。
2. **Given** 管理员在调用关系列表页面，**When** 通过部署子系统编号、调用发布单元编号、被调用发布单元编号、调用协议等字段查询，**Then** 系统返回匹配的调用关系列表。
3. **Given** 某条调用关系的被调用发布单元已被删除，**When** 系统渲染逻辑部署架构图时，**Then** 该调用关系不展示，但历史记录保留或由管理员手动清理。

### US-005 - 用户角色及权限管理

作为超级管理员，我希望设置管理员并管控权限，以便不同角色按需访问系统功能。

**验收场景**：
1. **Given** 超级管理员已登录系统，**When** 在用户管理页面为某个用户分配“管理员”角色，**Then** 该用户立即获得系统清单、技术栈、逻辑部署架构图的增删改查权限。
2. **Given** 普通用户已登录系统，**When** 访问系统清单查询页面，**Then** 系统允许查询；当该用户尝试进入新增或编辑页面时，**Then** 系统拒绝访问并提示无权限。

### 边界情况

- 当 Excel 导入的数据存在格式错误（如必填字段为空、枚举值非法）时，系统应记录错误行及原因，跳过错误行并继续导入合法数据，最终返回导入成功数与失败明细。
- 当管理员删除技术栈时，若该技术栈已被发布单元引用，系统应提示引用关系，由管理员决定是否解除引用或取消删除。
- 当自动生成架构图时，若某部署子系统下暂无发布单元，系统应展示空白图并提示“该部署子系统下暂无发布单元，请先维护发布单元数据”。
- 当两个发布单元之间存在多条调用关系（不同协议）时，架构图应分别展示各条调用关系及其协议标注。

## 三、功能性需求

### 3.1 系统清单管理
- **FR-001 [US-001]**：系统必须支持对“系统”实体进行增删改查，字段要求见数据实体章节，查询条件包括系统主数据编号、系统编号、中文名称、英文名称、类型、属主、系统级别、等保级别等。
- **FR-002 [US-001]**：系统必须支持对“子系统”实体进行增删改查，查询条件包括系统编号、子系统编号、子系统主数据编号、中文名称、英文名称、业务部门、子系统级别等。
- **FR-003 [US-001]**：系统必须支持对“部署子系统”实体进行增删改查，查询条件包括系统编号、子系统编号、子系统主数据编号、部署子系统编号、中文名称、英文名称等。
- **FR-004 [US-001]**：系统必须支持对“发布单元”实体进行增删改查，查询条件包括系统编号、子系统编号、子系统主数据编号、部署子系统编号、发布单元编号、中文名称、英文名称、类型、部署方式等。
- **FR-005 [US-001]**：系统必须支持系统、子系统、部署子系统、发布单元的层级关系维护：一个系统包含多个子系统，一个子系统包含多个部署子系统，一个部署子系统包含多个发布单元。
- **FR-006 [US-001]**：系统在执行删除操作时，若被删除对象存在下级关联数据（如系统存在子系统、子系统存在部署子系统、部署子系统存在发布单元），必须禁止删除并给出明确提示，要求用户先清理下级数据。
- **FR-007 [US-001]**：系统必须支持系统、子系统、部署子系统、发布单元的 Excel 批量导入。导入时，若数据按唯一编号已存在，则覆盖更新；若不存在，则新增。导入结果需返回成功数、失败数及失败明细。
- **FR-008 [US-001]**：系统必须支持按当前查询条件导出系统、子系统、部署子系统、发布单元的查询结果为 Excel 文件。
- **FR-009 [US-001]**：系统必须为 Excel 导入提供标准模板下载功能，模板中应包含必填字段标识及枚举值说明。

### 3.2 技术栈管理
- **FR-010 [US-002]**：系统必须支持对“技术栈”实体进行增删改查，查询条件包括技术栈编号、分类、名称、选型建议、负责条线等。
- **FR-011 [US-002]**：系统必须支持技术栈的 Excel 批量导入，覆盖更新规则同 FR-007，并提供标准模板下载。
- **FR-012 [US-002]**：系统必须支持按当前查询条件导出技术栈查询结果为 Excel 文件。
- **FR-013 [US-002]**：系统必须支持发布单元可选关联技术栈。在维护发布单元时，管理员可选择零个或多个技术栈进行关联；删除技术栈时，若存在关联的发布单元，系统应提示引用关系。

### 3.3 逻辑部署架构图管理
- **FR-014 [US-003]**：系统必须支持基于部署子系统下的发布单元、网络区域及发布单元调用关系，自动生成 Mermaid 格式的逻辑部署架构图。
- **FR-015 [US-003]**：逻辑部署架构图中必须展示网络区域（如办公网、互联网、DMZ、IPZ、CORE、边缘设备区域等）、发布单元名称、发布单元调用关系及调用协议。
- **FR-016 [US-003]**：架构图中当前系统发布单元必须以绿色（#90EE90）标识，其它系统发布单元必须以浅绿色（#E6F7E6）标识。
- **FR-017 [US-003]**：系统必须支持管理员对自动生成的架构图进行手动调整，包括调整节点位置、增加备注说明；手动调整结果需持久化保存，并在后续展示时优先使用已保存的手动调整结果。
- **FR-018 [US-003]**：当部署子系统下不存在发布单元时，系统应展示空白架构图并给出友好提示。

### 3.4 发布单元调用关系管理
- **FR-019 [US-004]**：系统必须支持对发布单元调用关系进行增删改查。调用关系字段包括：部署子系统编号、调用发布单元编号、被调用发布单元编号、调用协议（如 HTTP、TCP 等）。同一部署子系统下，调用发布单元与被调用发布单元组合作为联合唯一键。
- **FR-020 [US-004]**：系统必须支持按部署子系统编号、调用发布单元编号、被调用发布单元编号、调用协议等字段查询调用关系。
- **FR-021 [US-004]**：系统在执行调用关系删除时，仅删除关系记录，不影响发布单元实体本身。

### 3.5 用户角色及权限管理
- **FR-022 [US-005]**：系统必须支持三种角色：超级管理员、管理员、普通用户。
- **FR-023 [US-005]**：超级管理员必须拥有系统所有权限，并能够设置其他用户为管理员。
- **FR-024 [US-005]**：管理员必须拥有对系统清单、技术栈、逻辑部署架构图、发布单元调用关系的增删改查权限。
- **FR-025 [US-005]**：普通用户必须拥有对系统清单、技术栈信息、子系统发布单元、子系统逻辑部署架构图的查询权限，禁止执行任何新增、编辑、删除操作。
- **FR-026 [US-005]**：系统必须在前端页面和后端接口两个层面进行权限控制，确保普通用户无法绕过前端直接调用管理员接口。

## 四、数据实体

### 4.1 核心数据实体

- **系统（System）**：企业的顶层 IT 系统，是逻辑管理概念。
- **子系统（SubSystem）**：隶属于系统，为最终用户或其他系统提供具体功能。
- **部署子系统（DeploySubSystem）**：子系统下的逻辑部署概念，一套完整部署为一个部署子系统（如信创环境部署）。
- **发布单元（ReleaseUnit）**：物理上可执行的部署单元，如 war、jar、exe 等，归属于部署子系统。
- **技术栈（TechStack）**：企业的技术选型规范，包括分类、名称、选型建议等。
- **发布单元调用关系（ReleaseUnitRelation）**：记录发布单元之间的调用关系及协议。
- **用户（User）**：系统登录用户，关联角色权限。
- **角色（Role）**：超级管理员、管理员、普通用户。
- **网络区域（NetworkZone）**：部署架构图中的网络区域枚举（INTERNET、INTRANET、DMZ、IPZ、CORE、PTR 等）。

**实体关系**：
- 系统 与 子系统 是一对多关系。
- 子系统 与 部署子系统 是一对多关系。
- 部署子系统 与 发布单元 是一对多关系。
- 发布单元 与 技术栈 是多对多关系（可选关联）。
- 部署子系统 与 发布单元调用关系 是一对多关系。
- 用户 与 角色 是多对一关系。

## 五、非功能性需求

| 编号 | 类别 | 内容 | 来源 |
| --- | --- | --- | --- |
| NFR-001 | 用户规模 | 系统预估承载用户量为 100 人。 | 默认 |
| NFR-002 | 响应时间 | 普通查询操作响应时间应小于 2 秒；复杂操作（如批量导入、架构图生成）响应时间应小于 5 秒。 | 默认 |

| NFR-005 | 安全合规 | 请求应通过 HTTPS 传输，具备常规 SQL 注入、XSS 防护能力；系统涉及等保字段，需满足常规安全实践。 | 默认 |
| NFR-006 | 可用性 | 系统默认允许计划内停机，不要求高可用架构。 | 默认 |

## 六、项目约束与限制

- 系统交付形态为 **Web 管理后台**，通过浏览器访问使用。
- 逻辑部署架构图初版基于 **Mermaid** 格式渲染展示，后期可优化为自定义图形引擎。
- 系统、子系统等实体的部分枚举属性（如系统类型、系统级别、等保级别、子系统状态等）依赖数据字典维护，需在系统内提供数据字典管理或约定为硬编码枚举。

## 七、附录

### 需求澄清记录

#### 第1轮澄清
- 问题1: 该架构管理系统的交付形态是什么？
  用户回答：Web 管理后台（浏览器访问）
- 问题2: 删除系统、子系统或部署子系统时，对其下级数据的处理规则是？
  用户回答：限制删除：若存在下级数据，禁止删除并提示用户先手动清理下级数据
- 问题3: Excel 导入时，若数据已存在，处理策略是？
  用户回答：覆盖更新：用 Excel 中的数据覆盖系统中已存在的数据
- 问题4: 逻辑部署架构图的展示数据来源和维护方式是？
  用户回答：自动+手动：基础图形根据关系自动生成，但支持管理员手动调整节点位置、增加备注说明
- 问题5: 技术栈信息与发布单元之间是否需要建立关联关系？
  用户回答：可选关联：发布单元可以选填其使用的技术栈，但不做强制要求

#### 默认处理项
- 登录认证方式：默认采用系统自有账号密码登录，由超级管理员分配账号和角色；若后续需集成 SSO/LDAP，可作为二期需求。
- 导入导出数据量：默认按中小规模处理，采用同步导入导出方式；若后续数据量超过万级，可升级为异步任务队列。
- 操作日志：按最佳实践默认记录关键增删改操作日志，保留 90 天。
- 异常处理：按最佳实践默认处理，如参数校验失败返回明确错误信息、系统异常返回统一错误页面等。
- 子系统状态流转：默认由管理员手动变更状态（待上线 -> 已上线 -> 已下线），不设计自动流转规则。

#### 按需跳过的检查项
- 外部系统集成：用户原始需求中提及主数据平台编号，但未明确要求与主数据平台进行实时集成或同步，因此暂按“仅记录编号字段”处理，跳过外部集成详细设计。
- 存量系统影响面：本系统为新建系统，无存量改造影响面，跳过该项检查。
- 关键实现约束：用户未提出特殊技术约束或性能指标要求，按默认值处理。
